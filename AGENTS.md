# AGENTS.md

本文件用于约束 Codex 在本仓库中的默认工作方式。除非用户明确提出不同要求，否则均应遵守以下规范。

## 工作原则

- 优先做小而清晰的改动，避免一次性引入大范围、低收益的重构。
- 优先复用现有实现、已有工具函数、既有服务层能力与公共模型，避免重复开发相似功能。
- 修改前先阅读相关代码路径，确认仓库内是否已经存在可复用的实现。
- 新增代码必须与现有项目风格保持一致，避免引入不必要的抽象层。

## 测试优先

- 对于新需求、缺陷修复、公共行为变更，优先先编写或补充单元测试，再进行实际开发。
- 优先为新增或修改的“行为”编写测试，而不是只为实现细节写测试。
- 若当前模块已有测试模式，必须沿用现有测试组织方式。
- 若当前仓库缺少测试基础设施，或需求不适合先写单元测试，应先说明原因，再补充最小可行验证。
- 完成开发后，优先运行与本次改动直接相关的最小测试集，再视情况扩大验证范围。

## Public 接口与文档

- 当新增、修改或删除 public 接口时，必须同步更新 `README.md`。
- public 接口包括但不限于：HTTP API、CLI 参数、配置项、对外暴露的服务方法、事件/消息格式。
- `README.md` 中至少应同步更新以下内容：接口用途、关键参数、返回格式、调用示例。

## 复用优先

- 实现功能前，先检查是否已有相同或相近能力可复用。
- 如果已有实现仅需少量扩展，应优先扩展原实现，而不是新增一套平行逻辑。
- 避免复制粘贴已有函数后再局部修改；应优先抽取公共逻辑。
- 同类能力的入口、命名、返回结构应保持一致。

## 代码质量

- 禁止编写 very long 的面条式函数。
- 函数应保持单一职责，尽量短小、可测试、可复用。
- 复杂流程应拆分为多个清晰的私有辅助函数或服务方法。
- 避免在单个函数中混合路由处理、参数校验、业务编排、数据访问、格式转换等多种职责。
- 若函数已经明显过长，应优先考虑重构后再继续叠加逻辑。

## Python 异步开发规范

- 对于 FastAPI 的异步路由、异步服务方法与异步依赖，优先保持完整 async 调用链，优先使用异步 SDK、异步 HTTP 客户端、异步数据库客户端等实现。
- 禁止在 `async def` 中直接调用阻塞函数或阻塞式 I/O，包括但不限于同步 HTTP 请求、同步数据库访问、`time.sleep()`、阻塞式文件/网络读写等会卡住事件循环的操作。
- 若三方能力仅提供同步接口，应先评估是否已有可复用的异步实现；确需使用同步能力时，必须显式隔离其影响，避免直接阻塞主事件循环，并在实现说明中交代原因。
- 新增或修改 Python 集成代码时，必须先确认所依赖的 SDK 是否提供异步接口；如已提供异步接口，不应退回到同步写法。
- 设计异步流程时，应同时检查上游路由、中间服务与下游客户端是否一致支持 `await`，避免局部异步、整体阻塞的伪异步实现。

## 配置与环境变量

- 必要参数优先从 `.env` 或配置模块读取，不要在代码中写死。
- 包括但不限于：密钥、Token、数据库连接、服务地址、模型名、端口、开关类配置。
- 禁止在代码中硬编码敏感信息。
- 若某配置需要默认值，应在配置层集中定义，并说明用途；不要将默认值分散写在多个文件中。

## 实施要求

- 开发前先确认需求影响范围：路由、服务、数据模型、配置、测试、文档。
- 提交实现时，优先修复根因，不做表面补丁式修改。
- 非用户要求，不主动修改无关模块或顺手重构大面积代码。
- 如发现现有结构阻碍测试或复用，应做最小必要重构，并确保行为不变。

## 交付检查清单

在完成任务前，Codex 应尽量确认以下事项：

- 是否已优先补充单元测试。
- 是否已复用现有实现，而非重复开发。
- 是否避免引入超长、难维护函数。
- 是否避免在异步方法中使用阻塞函数或阻塞式 I/O。
- 是否将必要配置迁移到 `.env` 或统一配置入口。
- 是否在 public 接口变更后同步更新 `README.md`。
- 是否完成与改动直接相关的最小验证。
